# ssv — SystemVerilog Parameter Passing Core (Minimal Copyable Package)

This directory contains only the **core** of ssv: it dispatches the parameter
object tree described by `.cfg` to the right places. The full UVM template
(sequencer / driver / monitor / agent / env / test) and placeholder interfaces
are **not** here — those are project-specific templates provided by each
project's own `tb/env/` and `tb/tests/`.

---

## What's in this directory

```
ssv/
├── ssv_object.sv     // virtual class: dispatch_path + apply_overrides_from_file
├── ssv_pkg.sv        // wraps ssv_object into package ssv_pkg
└── README.md         // this file
```

Only three things. `ssv_object.sv` is the truly indispensable core;
`ssv_pkg.sv` is just a wrapper so you can `import ssv_pkg::*`.

---

## How to move it into another project

1. **Copy this directory**:
   ```bash
   cp -r <orig>/tb/utiles/ssv/  <new>/tb/utiles/ssv/
   ```
   (You can also name it `vendor/ssv/` or anything you like, as long as it
   ends up on the include path.)

2. **Add it to the project's filelist.f** (must come before your cfg package):
   ```
   +incdir+tb/utiles/ssv
   tb/utiles/ssv/ssv_pkg.sv

   # then your cfg package (ssv_cfg_pkg generated by cfg2sv.py)
   +incdir+gen
   gen/ssv_cfg_pkg.sv
   ```

3. **Copy the scripts** into `<new>/tools/`:
   * `tools/cfg2sv.py` — converts `<new>/cfg/*.cfg` into `<new>/gen/*.sv`
   * `tools/run_case.py` — runs simulations per `<new>/cases/*.cmd`

4. **Write your own cfg / cmd** (following the `.cfg` syntax and `.cmd`
   syntax), then write your own UVM test base (see `tb/tests/my_test.sv` in
   this project for the shape: in `build_phase` call
   `ssv_object::apply_overrides_in_cwd(root_cfg)`).

## How to use ssv_object (in user test)

There is exactly one real usage site: in `build_phase` call
`ssv_object::apply_overrides_in_cwd(root_cfg)`. Internally this class opens
the relative path `"cfg.info"` (relies on cwd = case run dir, set by
Python `run_case.py` in `subprocess.run(cwd=case_run_dir)`). The Python side
extracts the `ssv_cfg:` lines from `cases/<case>.cmd`, expands globs via
Python `fnmatch`, and writes the result to `cfg.info` — **SV does not read
cases/*.cmd, does not read plusarg/env, and does not do glob expansion**.

```sv
package my_test_pkg;
  `include "uvm_macros.svh"
  import uvm_pkg::*;
  import ssv_pkg::*;
  import ssv_cfg_pkg::*;   // cfg package generated by cfg2sv
  import my_env_pkg::*;    // user env package

  class my_test extends uvm_test;
    `uvm_component_utils(my_test)

    my_env        env;
    ssv_root_cfg  root_cfg;   // auto-generated root cfg class (from cfg/top.cfg)

    virtual function void build_phase(uvm_phase phase);
      super.build_phase(phase);

      root_cfg = ssv_root_cfg::type_id::create("ssv_root_cfg");

      // ★ ssv's only public entry point: opens the relative path "cfg.info"
      //   (relies on cwd = case run dir, set by Python run_case.py).
      ssv_object::apply_overrides_in_cwd(root_cfg);

      uvm_config_db#(ssv_object)::set(null, "uvm_test_top.env", "cfg", root_cfg);
      env = my_env::type_id::create("env", this);
    endfunction

    virtual task main_phase(uvm_phase phase);
      phase.raise_objection(this);
      `uvm_info("MY_TEST", $sformatf("NUM_AGENTS=%0d", root_cfg.NUM_AGENTS), UVM_LOW)
      #100ns;
      phase.drop_objection(this);
    endtask
  endclass
endpackage
```

The only thing the caller really uses is the single static method
`ssv_object::apply_overrides_in_cwd(root_cfg)`. Together with the
`apply_field` / `find_sub` hooks auto-generated by `cfg2sv.py`, it walks the
hierarchy down to each concrete leaf field.

> Other APIs inside `ssv_object` (for reference / advanced scenarios only):
> - `apply_overrides_from_file(root, path)`: read an `ssv_cfg:` file from any path
> - `apply_overrides(root, path)`: alias for `apply_overrides_from_file`
> - The old `apply_overrides_from_cmd` / `apply_overrides_from_pwd` /
>   `path_is_glob` / `ssv_wildcard_match*` have been removed — passing the
>   run dir is done by Python via cwd, and all glob expansion is done on the
>   Python side.

## What does the generated cfg class look like

After `cfg2sv.py` runs, `gen/<class_name>.sv` looks like this (excerpt):

```sv
class my_root_cfg extends ssv_object;
  rand int unsigned NUM_AGENTS = 2;     // default value
  string ENV_NAME = "default";
  sub_cfg agent_a;                       // sub-object handle

  `uvm_object_utils_begin(my_root_cfg)
    `uvm_field_int(NUM_AGENTS, UVM_DEFAULT)
    `uvm_field_string(ENV_NAME, UVM_DEFAULT)
    `uvm_field_object(agent_a, UVM_DEFAULT)
  `uvm_object_utils_end

  function new(string name = "my_root_cfg");
    super.new(name);
    agent_a = sub_cfg::type_id::create("agent_a");
    agent_a.AGENT_ID = 0;                // per-instance override
  endfunction

  // --- virtual hooks on ssv_object, auto-generated by cfg2sv ---
  function bit apply_field(string field_name, string value);
    case (field_name)
      "NUM_AGENTS": begin NUM_AGENTS = ssv_object::ssv_atoi(value); applied = 1; end
      "ENV_NAME":   begin ENV_NAME   = ssv_object::ssv_unquote(value); applied = 1; end
      ...
    endcase
    return applied;
  endfunction

  function ssv_object find_sub(string inst_name);
    case (inst_name)
      "agent_a": return agent_a;
      ...
    endcase
  endfunction
endclass
```

## Known Limits

- Numeric field overrides only parse decimal digits / `0` / `1` / `1'b0` /
  `1'b1`. For other complex literals (hex/bin), write the default value
  directly in the `.cfg`.
- Sub-object blocks inside `.cfg` files do not allow further nested
  sub-objects (it's a tree, not a DAG).
- Validated on VCS W-2024 / UVM 1.2; other simulators may need include /
  DPI-link flag tweaks.
